# PR Review Skill

PRのコードレビューを自動実行し、GitHubにレビューコメントを投稿します。

## 入力

ユーザーから以下のいずれかで指定される：
- PR番号（例: `#1234`, `1234`）
- PR URL

## 実行フロー

### 1. PR情報の取得
```bash
gh pr view <番号> --json title,body,baseRefName,headRefName,files,additions,deletions
gh pr diff <番号>
```

### 2. 変更ファイルの分析
- 変更ファイル一覧と差分を確認
- 変更の目的・スコープを理解

### 3. レビュー観点
以下の観点でレビューする（**簡潔に、要点のみ**）：

- **正確性**: ロジックのバグ、エッジケースの見落とし
- **スコープ**: 不要な変更が含まれていないか（最小限の変更か）
- **フォーマッタ起因の不要 diff**: rubocop / prettier 等の自動整形のみの変更が混入していないか確認し、検出した場合はフラグする
- **セキュリティ**: SQL Injection、XSS、認証漏れ等
- **パフォーマンス**: N+1、不要なクエリ、メモリ問題
- **テスト**: テストの有無・網羅性

### 4. レビュー投稿
```bash
gh pr review <番号> --comment --body "<レビュー内容>"
```

- **`--approve` は使わない**（自分のレビューで PR を approve しない）
- 必ず `--comment` でコメント投稿する

## レビュー出力フォーマット

```markdown
## レビュー

### 総評
（1-2文で変更の評価）

### 指摘事項
- [重要度] ファイル:行番号 - 指摘内容

### Good
- 良い点があれば簡潔に
```

## 重要ルール

- **簡潔に書く**: 冗長な説明は不要。指摘は端的に
- **最小スコープを尊重**: 「ここもリファクタしたほうが」等の余計な提案はしない
- **推測で指摘しない**: コードを確認して根拠のある指摘のみ行う
- **speculative なリスクを書かない**: フレームワークの既定挙動・一般論・「念のため」レベルの懸念は指摘欄に書かない。実在する・再現するリスクのみ書く
- **self-approve しない**: 自分で作成・実装した PR を `--approve` しない。レビューは必ず `--comment` で投稿
- レビュー投稿前にユーザーに内容を確認してもらう
